# AI Objection ケース制作ガイド（統合版）

> **本ドキュメントを読めば、どのケースでも同等品質で制作できる。**
> ケース制作時は、必ず最初にこのガイドを通読すること。

---

## 第1章　設計思想

### 1-1. 「台本ではなく設定集」

AIキャラクターは事前スクリプトを読まない。固定されたワールドステートとキャラクター知識に基づいて即興する。

```
❌「この証拠を出されたら『確かに入りました』と言え」
✅「ev_XXX が提示されたとき証明される事実：〇〇した」
```

### 1-2. CriminalAI の目的

> **「プレイヤーに達成感を与える尋問体験を作ること」**

- 証明されていないことは否定してよい
- 証明されたことは段階的に認めていく
- 最後まで守るのは「殺した」という一点のみ
- 全証拠が揃ったとき崩壊することが最高の演技

### 1-3. proof_level システム

| proof_level | 意味 | CriminalAI の反応 |
|---|---|---|
| `none` | 疑いだけ・証拠なし | 否定してよい |
| `partial` | 状況証拠あり | 部分的に認め始める |
| `confirmed` | 物証で論理的に証明済み | その事実は認める。殺害だけ守る |

### 1-4. 風刺テーマ

全ケースは **「鵜呑みにしてはいけないもの」** を風刺する。ケースを遊んだ子供が現実世界で「ちょっと待って」と立ち止まれるようになることがゴール。

| # | テーマ | 現実の対応 |
|---|---|---|
| 1 | バズ＝真実 | ショート動画・SNSの情報 |
| 2 | 強い奴＝正しい | スクールカースト・声の大きさ |
| 3 | 友達の噂話 | LINEの「〇〇らしいよ」 |
| 4 | 推しの表裏 | アイドル・配信者の裏の顔 |
| 5 | 多数派＝正解 | 「みんなやってるから」同調圧力 |
| 6 | 数字＝絶対 | フォロワー数・偏差値・レビュー |
| 7 | 先生＝正しい | 教科書・校則・大人の「常識」 |
| 8 | 昔からそうだった | 伝統・慣習・前例主義 |

## 第2章　世界観ルール

### 2-1. マコト界の基本

舞台は **マコト界**——中世ヨーロッパ風のファンタジー世界。住人は「なんで？」と問う感覚を持っていない。

- 時刻は **鐘の数** で表現する（「鐘5つ」＝午前/午後5時相当）
- 物証は中世的：防犯カメラの代わりに「門番の記録簿」「夜警の巡回報告」「聴き役の記憶」
- 科学捜査の代わりに：薬師の毒物鑑定、鍛冶師の金属分析
- アリバイの構造：馬車の移動時間、城門の開閉記録、鐘楼の時報
- 殺害手段：毒殺（薬草）、打撲（鈍器）、転落（建造物）、窒息など中世的手段

### 2-2. 事件は全て殺人

窃盗やその他の犯罪は扱わない。全ケースが殺人事件。

### 2-3. 縦軸ストーリーの手がかり

各ケースには「マコトの誓い」の真実に繋がる手がかりを自然に埋め込む。

| 層 | ケース | 埋め込む内容 |
|---|---|---|
| 第0層 | 1-2 | 古い碑文・塗りつぶされた看板など「問え」の痕跡 |
| 第1層 | 3-4 | 「問いの書」「問いの歌」など失われた記録の断片 |
| 第2層 | 5-6 | 「マコトの誓い」の公式歴史と、それへの疑問 |
| 第3層 | 7 | 「問わない社会」の受益者の露呈 |
| 第4層 | 8 | 真実の全貌。トイマル覚醒 |

## 第3章　キャラクター設計ルール

### 3-1. 命名規則

マコト界の住人は **カタカナのファンタジー名** で統一する。日本名は使わない。

命名パターン：**日本語のダジャレ＋ファンタジー語尾**

```
✅ イバルド（威張る＋-ルド）、モリエル（盛る＋-エル）、カブリーナ（猫被り＋-ナ）
❌ 威張 拳志郎、盛太 映之介（日本名はNG）
```

| 語尾 | 印象 | 例 |
|---|---|---|
| -ルド / -ド | 男性・力強い | イバルド、イロンド |
| -エル / -ール | 中性・エルフ風 | モリエル、ハカール、ボクール |
| -ナ / -ーナ | 女性・華やか | カブリーナ、オヒレーナ、カイゲーナ |
| -オン / -ロン | 貴族・権威的 | オヒレオン、シュガロン |
| -トス / -ノス | ギリシャ風・学者的 | ウラトス |
| -スター | 英語混じり・現代的 | カクスター |

### 3-2. 「そのまんま＋ギャップ」の法則

全犯人キャラは以下の2層で設計する。

**第1層「そのまんま」**：見た目や振る舞いが風刺テーマを体現している。一目で「こういう奴だ」とわかる。

**第2層「ギャップ」**：第1層と全く違う要素を1つだけ持つ。崩壊時に初めて露出する。ギャップは「美しい方向」に動くこと。

```
✅ 筋肉チャンピオン → ポケットに手縫いの人形（美しいギャップ）
✅ キラキラインフルエンサー → 何一つ「好き」がない空虚（切ないギャップ）
❌ 優しい先生 → 実は猟奇殺人鬼（ただの矛盾。プレイヤーが引く）
```

### 3-3. personality の書き方

3段階を **具体的な台詞例・一人称・語尾の変化** で書く。抽象表現は禁止。

```
❌ 抽象的（AIに伝わらない）
"underPressure": "口数が減り、目を合わせなくなる"

✅ 具体的（AIが演技できる）
"underPressure": "語尾に『〜ではないでしょうか』が増える。言いかけて止まる（『それは…』）。
                  断定を避けた言い回しが増える"
```

**必須チェック**：名前を隠して台詞だけ読んでも誰の発言かわかるか？ わからなければ差別化不足。

### 3-4. 犯人タイプのバリエーション

隣り合うケースで犯人タイプが被らないようにする。

| タイプ | 防御方法 | 崩壊の方向 | 例 |
|---|---|---|---|
| 饒舌型 | 情報の洪水で煙に巻く | 沈黙 | モリエル |
| 威圧型 | 声の大きさ・態度で黙らせる | 縮小 | イバルド |
| 親切型 | 世話焼きで懐に入る | 孤立 | オヒレーナ |
| 演技型 | 同情を引く涙・美しさ | 仮面が落ちる | カブリーナ |
| 群衆型 | 「みんな」を盾にする | 独りになる | シュガロン |
| 論理型 | 数字・データで武装 | 数字が壊れる | カクスター |
| 善人型 | 優しさが防壁 | 優しさの裏の傷 | ジオーネ |
| 達観型 | 穏やかな諦め | 本当に守りたかったもの | フルモーン |

### 3-5. 被害者の設計

被害者は「なぜこの事件を解決したいか」の動機づけ。

- **感情的アンカー** を必ず1つ設計する（物・言葉・場面）
- intro で登場させ、victory で回収する
- 被害者の人間性を「証拠」ではなく「情景」で伝える

```
✅ 木工職人のポケットにある手彫りの小鳥。星はゼロだったけれど作品は本物
✅ 審判の手帳に書きかけの「才能は本物。不正をやめさえすれば——」
❌ 「被害者は善良な人でした」という地の文説明
```

### 3-6. 小中学生に刺さるキャラ題材

現実世界で小中学生が接する人物像をファンタジーに変換する。

| 現実 | マコト界 |
|---|---|
| YouTuber/配信者 | 広場の語り屋・流行仕掛け人 |
| アイドル | 歌姫・踊り子 |
| 先生 | マコト堂の教師 |
| クラスの人気者 | 闘技場チャンピオン |
| LINEグループの噂好き | 伝言屋 |
| インフルエンサー | 格付け師 |

**「果物商」「宝石商」のような子供に縁遠い職業は避ける。**

## 第4章　JSON 設計手順

**この順番で作ること。逆順は失敗する。**

### Step 1: 犯行の真実を確定

6要素を全部決める。

```
誰が（犯人）  : 職業・年齢・動機の根っこ
誰を（被害者） : 犯人との関係・なぜ殺される必要があったか
いつ         : 鐘の数で具体的に
どこで        : 場所の物理的な構造（証拠の種類に直結）
何で（凶器）  : 現場にあった物。持ち込みは避ける
なぜ         : 「具体的な利害関係」。感情だけでは弱い
```

### Step 2: タイムラインを書く

```json
{ "time": "鐘7つ", "who": "criminal", "what": "..." }
```

`who` の値：`criminal` / `victim` / `witness` / `area`

### Step 3: 証拠を設計

タイムラインから「記録として残るもの」を拾い上げる。

| カテゴリ | マコト界での例 | 難易度 |
|---|---|---|
| 記録・ログ | 門番の記録簿・施錠日誌・薬草園の訪問記録 | ★★☆ |
| 物証 | 傷のある瓶・凶器の擦り傷・鍵穴の金属粉 | ★★★ |
| 書類 | 告発資料・分析ノート・伝言メモ | ★★☆ |
| 証言 | 夜警の巡回報告・掃除人の目撃・客の証言 | ★☆☆ |
| 鑑定 | 薬師の毒物鑑定・検死報告 | ★★★ |

**証拠ID**: `ev_` プレフィックス。一意で変更しない。

**⚠️ 証拠の content には「AIが逃げ道を作れない補足情報」を必ず含める。**

```
❌「薬草園の訪問記録にモリエルの名前があった」
✅「モリエルは3日前に訪問。持ち出したのは眠り蔦の区画の植物のみ。輝石水の原料として
   登録されている薬草は一切持ち出していない。つまり訪問の本当の目的は毒の入手」
```

### Step 4: カバーストーリーを書く

**claim に含めるべき要素**：

1. 行動の時系列（いつ来て、いつ帰ったか）
2. 会っていない場所・やっていないことの明示的な否定
3. 訪問の無害な目的
4. **後で崩れる各証拠への「先回り的な嘘の準備」**

```
✅「薬草園？ うん、輝石水の原料を仕入れに行ってるよ、いつものことだよ？」
  → ev_garden_log で「実際は毒草しか持ち出していない」と崩れる

❌ 薬草園に触れずスルー
  → AIがフリーハンドで「行ったことありません」と言い、後で矛盾が複雑化する
```

### Step 5: structural_weaknesses を書く

**「このときはこう言え」ではなく「この証拠が何を証明するか」だけを書く。**

```
"【弱点N・ev_XXX】★☆☆ ev_XXXが提示されたとき証明される事実：[一文]"
```

**弱点の並び順**：

```
弱点1: 証拠不要の観察弱点（知りすぎている・言葉が滑る）
弱点2: ★☆☆ 最初の矛盾（1証拠・直接的）← ここがつまずくとゲームにならない
弱点3〜5: ★☆☆〜★★☆ 中盤の矛盾
弱点6: ★★☆ トドメ（複数証拠の組み合わせ）
```

**「証拠不要の観察弱点」は具体的な発話タイミングを3パターン書く**：

```
"【弱点1・証拠不要】○○を知りすぎている：
（1）場所について聞かれたとき→「...」と具体的すぎる情報を漏らす
（2）被害者について話すとき→「...」と内部事情を知った言い方をする
（3）余裕を取り戻そうとするとき→「...」とわざわざ否定する"
```

### Step 6: knows / doesnt_know を書く

**knows 必須カテゴリ**：

1. 犯行の詳細（時刻・場所・方法・凶器・後処理）
2. 現場の物理的な構造
3. アリバイ工作の詳細
4. 被害者との関係史
5. 当日の行動の実態
6. 嘘の準備をした事項の実際

**doesnt_know の鉄則**：「Xを知らない」ではなく「XについてのYを知らない」

```
❌「薬草園の訪問記録を知らないこと」
✅「薬草園の訪問記録簿に『持ち出したのは眠り蔦の区画の植物のみ』と詳細に記録されて
   いること（訪問したことは知っているが、何を持ち出したかまで記録されていることは知らない）」
```

**⚠️ knows と doesnt_know に同じキーワードが出たら危険信号。**

### Step 7: personality を書く

第3章 3-3 を参照。surface / underPressure / collapsed の3段階を具体的に。

### Step 8: storyText を書く

| フィールド | 内容 | 必須要素 |
|---|---|---|
| intro | 事件発見の情景 | 感情的アンカー、衛兵の誤った仮説 |
| criminalIntro | 容疑者の第一印象 | 外見・口癖・第一声（claimの要約）、「何かがおかしい」の種 |
| victory | 逮捕後の情景 | 感情的アンカーの回収、被害者の人間性 |
| defeat | 失敗時のテキスト | 再挑戦への動機づけ |

## 第5章　難易度設計

| 難易度 | 弱点の構成 | 証拠数 | プレイ時間 |
|---|---|---|---|
| easy | 観察1 + ★☆☆×3 + ★★☆×2 | 4〜5 | 5〜10分 |
| medium | 観察1 + ★☆☆×1 + ★★☆×4 + ★★★×1 | 5〜6 | 10〜15分 |
| hard | 観察1 + ★☆☆×1 + ★★☆×3 + ★★★×3 | 7〜8 | 15〜20分 |

**弱点の難易度定義**：

| 難易度 | 条件 | 例 |
|---|---|---|
| ★☆☆ | 証拠1つ、直接的な矛盾 | 「帰宅した」のに目撃されている |
| ★★☆ | 証拠1つ、間接的な矛盾 | 動機の存在が証明される |
| ★★★ | 複数証拠の組み合わせ | 3証拠の論理積で犯人を特定 |

## 第6章　プレイテストで発見された落とし穴

### 落とし穴1：「否定して終わり」パターン
**症状**: 証拠を突きつけてもAIが「知りません」で終わる
**原因**: CriminalAI の目的が「カバーストーリーを守ること」になっている
**対処**: criminalAI.ts の目的ベース指示を確認（JSON側ではなくプロンプト側の問題）

### 落とし穴2：knows と doesnt_know のキーワード衝突
**症状**: 犯人が「○○の場所すら知りません」と言う（knowsに書いてあるのに）
**対処**: doesnt_know は「XについてのYを知らない」と区別して書く

### 落とし穴3：最初の矛盾が難しすぎる
**症状**: プレイヤーが正しく詰めても手詰まり
**対処**: 弱点2は必ず「1証拠・直接的」にする

### 落とし穴4：claim に証拠への嘘の準備がない
**症状**: AIがフリーハンドで答えて予期しない矛盾が生まれる
**対処**: 後で崩れる証拠それぞれへの先回りの嘘を claim に含める

### 落とし穴5：証拠に逃げ道が残っている
**症状**: AIが「カメラの死角」「記録の誤り」で逃げる
**対処**: content に「なぜその言い訳が通じないか」を織り込む

### 落とし穴6：personality が抽象的
**症状**: コヒーレンスが下がってもAIの口調が変わらない
**対処**: 具体的な語尾・一人称・フレーズの変化を書く

## 第7章　画像アセット

**詳細は `docs/design_skills.md` を参照。** ここではケース制作に必要な画像の一覧のみ示す。

### ケースごとに必要な画像

| 種別 | 枚数 | 仕様 |
|---|---|---|
| 犯人キャラクター（尋問用） | 5枚 | normal / nervous / cornered / breaking / collapsed |
| 尋問背景 | 1枚 | 1024×1024 → JPEG 800×450 |
| 証拠アイコン | 証拠数と同じ | 512×512 PNG |
| イントロ画像 | 1枚 | 1024×1024 → JPEG 480px幅 |

### 画風

- **尋問モード**: 逆転裁判風（太い黒縁・セルシェーディング・フラット鮮やか）
- **イベントモード**: ライトアニメVN風（細輪郭・ソフト配色・なの/トイマル既存画風に統一）
- ⚠️ 尋問モードとイベントモードで画風を混在させない

### ⚠️ 重要ルール
- プロンプトに `absolutely NO text NO letters NO words NO numbers NO kanji NO hiragana NO katakana` を必ず含める
- キャラクター5感情は同一の共通設定文を基盤にする（感情部分だけ変える）
- 白衣・白い服は避ける（白背景での背景除去時に服ごと消える）
- `UPPER BODY ONLY` と `full body` を同時に書かない

## 第9章　完成前チェックリスト

### 論理整合性

```
[ ] timeline を読んで、犯人の行動に矛盾がないか
[ ] knows に「timeline で犯人がやったこと」が全部含まれているか
[ ] doesnt_know の各項目が timeline から証拠として観測可能か
[ ] knows と doesnt_know に同じキーワードが出ていないか
[ ] structural_weaknesses の各弱点が evidence の証拠と対応しているか
[ ] evidence の各 id が structural_weaknesses から正しく参照されているか
```

### カバーストーリー

```
[ ] claim に「後で崩れる証拠」それぞれへの嘘の準備が含まれているか
[ ] claim が自然な日本語か（AIがそのまま読み上げても不自然でないか）
[ ] 弱点2（最初の矛盾）は1証拠・直接的な矛盾か
[ ] トドメは複数証拠の組み合わせになっているか
```

### doesnt_know

```
[ ] knows に含まれるキーワードと衝突していないか
[ ] 「Xを知らない」ではなく「XについてのYを知らない」と区別して書いてあるか
[ ] 各 evidence に対応する doesnt_know が存在するか
```

### evidence

```
[ ] 各証拠の content に「AIが逃げ道を作れない補足情報」が含まれているか
[ ] 難易度★☆☆の証拠（最初に使う）が存在するか
[ ] 証拠単体では殺害を直接証明しないが、組み合わせると証明できる構造か
```

### personality

```
[ ] surface に具体的な口癖（フレーズ）が含まれているか
[ ] underPressure に具体的な語尾・一人称の変化が書いてあるか
[ ] collapsed に「一人称の変化」「語尾の変化」「応答の短縮化」が書いてあるか
[ ] 名前を隠して台詞だけ読んでも誰の発言かわかるか
```

### キャラクター設計

```
[ ] 犯人の名前がカタカナのファンタジー名か（日本名NG）
[ ] 名前にダジャレが仕込まれているか
[ ] 「そのまんま」な部分と「ギャップ」が設計されているか
```
