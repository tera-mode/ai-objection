# 実装指示書：トイマル CompanionAI「聴き溜め」システム

> **対象リポジトリ**: `https://github.com/tera-mode/ai-objection`
> **関連ドキュメント**: `docs/world_concept.md`（トイマルの設定）、`docs/case_dev_guide.md`（ケース制作ガイド）、`docs/case001_request.md`、`docs/case002_request.md`

---

## 背景・目的

### 解決すべき問題

1. **証拠の出所が不自然**: case_001以降の本格ケースでは「門番の記録簿」「薬師の毒物鑑定書」など専門的な証拠が必要だが、なぜプレイヤーが最初からそれを持っているのか説明がつかない
2. **トイマルが尋問画面に不在**: 設計書のコアループには CompanionAI の存在が定義されているが、尋問画面には未実装
3. **証拠の説明文に主観が混入**: 現状の evidence.content に「つまり〜だった」等の解釈が含まれている。プレイヤー自身が考える余白を奪っている

### 設計思想

トイマルは「聴き役」族。一度聞いた言葉を一字一句記憶する。ただし「問う」回路がないため、情報の意味を理解できない。

マコト界は**口伝社会**。あらゆる記録（門番の入場者読み上げ、薬師の鑑定報告、夜警の巡回報告）は誰かが声に出して誰かに伝えている。トイマルは街を歩き回る中でそれらの声を全て耳で吸い込んでいる。

しかしトイマルにとってそれは「聞いた言葉の山」でしかない。**プレイヤーが犯人の発言からキーワードを拾い、トイマルに渡すことで、該当する記憶が浮上し、証拠としてアンロックされる。**

トイマルが持つのは「レコーダー」機能のみ。「矛盾を検出する」能力は持たない。聴いた事実をそのまま再生するだけ。そこから矛盾に気づき、犯人にぶつけるのはプレイヤーの仕事。

## Part 1：ゲームフロー（改訂版）

### 証拠ゼロスタート

```
事件概要の提示（CrimeSceneフェーズ）
  → 証拠ゼロ。事件の状況のみ知る
  → 証拠スロットが [ ? ][ ? ][ ? ][ ? ] と表示される（数だけわかる）
  ↓
尋問開始（Interrogationフェーズ）
  → 丸腰で犯人と話し始める
  ↓
  プレイヤーが犯人に質問
    → CriminalAI が返答（嘘を含む）
    → JudgeAI が矛盾判定 + コヒーレンス反映
    → 犯人の返答を元に、CompanionAI がキーワード候補チップを生成
    → プレイヤーがチップを選択 or 自由入力でトイマルに渡す
    → トリガー判定：ヒットすれば証拠アンロック演出
    → アンロックされた証拠は次のターンから犯人への突きつけに使用可能
  ↓
コヒーレンス 0 → 自動崩壊・逮捕成立
```

### ターン消費ルール

- **犯人への質問**: ターンを1消費する
- **トイマルへのキーワード渡し**: ターンを消費しない。何回でもできる
- **証拠の突きつけ**: ターンを1消費する（犯人への質問の一種として扱う）

#### キーワード候補チップの生成

犯人が返答するたびに、CompanionAI が「気になるキーワード候補」を2〜3個チップとして表示する。

- チップは犯人の直近の発言から抽出された名詞・場所・時間・行動を含む短いフレーズ
- チップは毎ターン更新される（犯人の最新の発言に基づく）
- プレイヤーはチップを選択する **or** 「+ 自分で入力」で自由テキストを渡せる
- チップの中にトリガーに該当するものが含まれている保証はない（ハズレチップもある）

#### ヒット時の演出

1. トイマルの目がパッと開く（表情変化 → `event_eureka` 等の新規表情）
2. セリフ表示：「あっ！ 思い出したのだ！」
3. **証拠スロットの [ ? ] が光って → パカッと開く演出**
4. 証拠の名前と「声」がテキスト表示される
5. パネルの「トイマルの記録」欄に正式登録
6. 対話ログにもトイマルの吹き出しとして表示

#### ミス時の演出

1. トイマルが首をかしげる
2. 「うーん……それについてはボクの耳に残ってないのだ」
3. ペナルティなし。すぐ再試行可能

## Part 3：case JSON スキーマの拡張

### 新規フィールド: `companionMemory`

`evidence` 配列と対になる形で、各証拠のアンロック条件をトイマルの「聴き溜め」として定義する。

```json
{
  "id": "case_001",
  "title": "流行通りの色水",
  "evidence": [
    {
      "id": "ev_analysis",
      "name": "品質検査人の声",
      "content": "『輝石水の成分を調べたのだけれど——薬効成分はゼロ。着色料と砂糖水だけだったわ。明日、衛兵に報告するつもり』——メキーラが同僚に話していた声"
    }
  ],
  "companionMemory": [
    {
      "id": "mem_analysis",
      "unlocks_evidence": "ev_analysis",
      "trigger_keywords": ["成分", "薬効", "効果", "色水", "偽物", "分析", "検査", "品質", "告発", "報告"],
      "toimaru_recall": "あっ！ 思い出したのだ！ 昨日、検査人のおばさんが同僚に話してた声が聞こえたのだ！",
      "source_description": "昨日の夕方、検査人の事務所の前を通りかかったとき"
    }
  ]
}
```

### フィールド定義

| フィールド | 型 | 説明 |
|---|---|---|
| `companionMemory[].id` | string | 一意のメモリID。`mem_` プレフィックス |
| `companionMemory[].unlocks_evidence` | string | アンロックする evidence の id |
| `companionMemory[].trigger_keywords` | string[] | CompanionAI がマッチング判定に使うキーワード群。完全一致ではなく意味的な近さで判定する |
| `companionMemory[].toimaru_recall` | string | ヒット時にトイマルが言うセリフ |
| `companionMemory[].source_description` | string | トイマルがいつどこでその声を聴いたかの簡潔な説明 |

### evidence.content の書き方ルール（変更）

**旧ルール**: 事実＋解釈＋AIが逃げられない補足情報を含める

**新ルール**: トイマルが聴いた「声」のみ。主観・解釈・結論を一切含めない

```
❌ 旧（主観あり）
"content": "モリエルは3日前に訪問。持ち出したのは眠り蔦の区画の植物のみ。
 輝石水の原料として登録されている薬草は一切持ち出していない。
 つまり、輝石水の原料調達は口実で、実際の目的は眠り蔦の入手だった。"

✅ 新（声のみ）
"content": "『3日前のモリエルさんの持ち出し記録ですね。
 眠り蔦の区画から1束。他の区画からの持ち出しはありません。
 輝石水の原料として登録されている薬草の持ち出し記録はゼロです』
 ——薬草園の管理人が衛兵に報告していた声"
```

ポイント:
- 全ての証拠は **「誰かが声に出して誰かに伝えた言葉」** として表現する
- 「つまり〜」「したがって〜」「これは〜を意味する」のような解釈は書かない
- AIが逃げ道を作れない補足情報は、**声の中に事実として自然に含める**（「他の区画からの持ち出しはありません」等）
- 末尾に「——誰が誰に言っていた声」の出典を記載する

## Part 4：CompanionAI の実装

### 概要

CompanionAI は以下の2つの機能を持つ。

1. **キーワード候補チップの生成**: 犯人の返答テキストから、プレイヤーがトイマルに渡しそうなキーワード候補を2〜3個抽出する
2. **トリガー判定**: プレイヤーが渡したキーワードと `companionMemory[].trigger_keywords` の意味的マッチングを行い、ヒット/ミスを判定する

### API エンドポイント

#### POST /api/companion-chips

犯人の返答後に呼ばれ、キーワード候補チップを返す。

リクエスト:
```json
{
  "caseId": "case_001",
  "criminalResponse": "ウチは朝から店の準備で忙しくて、メキーラの店なんか近づいてもないって",
  "unlockedEvidenceIds": ["ev_analysis"],
  "conversationContext": "直近3ターンの会話要約"
}
```

レスポンス:
```json
{
  "chips": ["朝の準備", "近づいていない", "メキーラの店"],
  "toimaruComment": "この人、すごくたくさん話すのだ……"
}
```

#### POST /api/companion-trigger

プレイヤーがキーワードを渡した時に呼ばれ、証拠のアンロック判定を行う。

リクエスト:
```json
{
  "caseId": "case_001",
  "playerKeyword": "朝の準備",
  "unlockedEvidenceIds": ["ev_analysis"],
  "conversationHistory": "犯人がこれまでに言った内容の要約"
}
```

レスポンス（ヒット）:
```json
{
  "hit": true,
  "unlockedEvidenceId": "ev_garden_log",
  "toimaruLine": "あっ！ 思い出したのだ！ 薬草園の管理人さんが衛兵に報告してた声が聞こえたのだ！",
  "sourceDescription": "今朝、薬草園の前を通りかかったとき"
}
```

レスポンス（ミス）:
```json
{
  "hit": false,
  "toimaruLine": "うーん……それについてはボクの耳に残ってないのだ"
}
```

### CompanionAI プロンプト設計

#### チップ生成プロンプト

```
あなたはファンタジー推理ゲームの相棒キャラ「トイマル」の思考を担当するAIです。

【あなたの役割】
犯人の返答テキストから、プレイヤーが「気になるかもしれない」キーワードやフレーズを
2〜3個抽出してください。

【抽出ルール】
- 名詞、場所名、時間、具体的な行動を含むフレーズを優先する
- 犯人が「否定した事実」は有力な候補（例：「近づいていない」「知らない」）
- 犯人が具体的すぎる情報を漏らした箇所も有力な候補
- 全てのチップがトリガーにヒットする必要はない（ハズレも混ぜてよい）
- すでにアンロック済みの証拠に関連するキーワードは除外する

【出力形式】
JSON配列で短いフレーズを2〜3個返す。説明は不要。
```

#### トリガー判定プロンプト

```
あなたはファンタジー推理ゲームの証拠アンロック判定を行うAIです。

【ルール】
プレイヤーが渡したキーワードが、以下の未アンロック証拠のトリガーキーワードと
意味的に関連しているかを判定してください。

完全一致は不要です。以下の基準で判定してください：
- プレイヤーのキーワードがトリガーキーワードの同義語・類語・関連語であればヒット
- プレイヤーのキーワードがトリガーキーワードの上位概念・下位概念であればヒット
- 全く無関係な場合のみミス

【重要】
- 迷ったらヒットにする（プレイヤーの努力を報いる方向に倒す）
- ただし、犯人がまだ関連する話題に一切触れていない場合はミスにする
  （犯人の発言がトリガーの文脈を含んでいることが前提条件）

【未アンロック証拠のトリガー一覧】
${未アンロックのcompanionMemoryをJSON形式で列挙}

【プレイヤーが渡したキーワード】
"${playerKeyword}"

【犯人がこれまでに話した内容】
${conversationHistory}

【出力】
ヒットした場合: { "hit": true, "matched_memory_id": "mem_xxx" }
ミスの場合: { "hit": false }
```

### 実装上の注意点

- **トリガー判定の前提条件**: 犯人がそのキーワードに関連する話題をすでに発言していること。犯人がまだ何も言っていない段階でプレイヤーが先回りしてキーワードを渡しても、ミスにする。これにより「犯人から情報を引き出す → トイマルに渡す」の順序が保証される
- **同一証拠の二重アンロック防止**: `unlockedEvidenceIds` で既にアンロック済みの証拠を渡し、判定対象から除外する
- **チップ生成の頻度**: 犯人の返答があった後、毎回チップを更新する。トイマルパネルを開いた時に最新のチップが表示される

| 旧 evidence.name | 新 evidence.name | 声の出所 |
|---|---|---|
| ev_lock_log（施錠記録） | 管理人の声 | 管理人が衛兵に「施錠記録」を読み上げた声 |
| ev_witness（伝言の証言） | 弟子の声 | 弟子が「イバルドからハカールへの伝言を預かった」と衛兵に話した声 |
| ev_injury_report（検死報告） | 薬師の声 | 薬師が「打撲の角度が組み手では起こりえない」と報告した声 |
| ev_forbidden_tech（禁止技資料） | ハカールの声 | ハカールが同僚の審判に「告発の準備をしている」と話していた声 |

### companionMemory のトリガー設計方針

- イバルドは「威圧型」犯人。短い言葉で突っぱねるため、犯人の発言からキーワードを拾いにくい
- よって、trigger_keywords は幅広く設定する
- 例：イバルドが「俺は鐘7つに帰った」と言えば → 「鐘7つ」「帰った」「帰宅」あたりでトリガー → 施錠記録の声がアンロック

※ case_002 の JSON 詳細は、case_002 の制作指示書（`docs/case002_request.md`）の証拠設計を参照し、上記ルールで変換すること。

## Part 7：CriminalAI への証拠突きつけフローの変更

### 現状

プレイヤーは最初から全証拠を閲覧でき、質問テキスト内で証拠に言及することで突きつける。

### 変更後

- プレイヤーはアンロック済みの証拠のみ閲覧・突きつけ可能
- 証拠パネルのUI上で、アンロック済みの証拠をタップ →「この証拠を突きつける」ボタン → テキスト入力欄に証拠参照が自動挿入される仕組み（任意。テキストで直接言及してもよい）
- JudgeAI は、プレイヤーの質問に含まれる証拠がアンロック済みかどうかを `unlockedEvidenceIds` でチェックする。未アンロックの証拠への言及は `proof_level: "none"` として扱う

### CriminalAI プロンプトへの追加

`buildCriminalAIPrompt` の systemPrompt に以下を追加:

```
【証拠の状態】
現在プレイヤーがアンロック済みの証拠: ${unlockedEvidenceIds.join(', ') || 'なし'}
プレイヤーがまだアンロックしていない証拠の内容は、
プレイヤーが言及しても「何の話だ？」と返してよい。
```

## Part 8：トイマルの尋問中リアクション

### 犯人の返答後の短いコメント

CompanionAI は犯人の返答後、対話ログに短いトイマルコメントを表示する（1〜2文）。
これはチップ生成APIの `toimaruComment` として返される。

| 状況 | コメント例 |
|------|-----------|
| 犯人が余裕で否定 | 「この人、すごく自信があるのだ……」 |
| 犯人がやや動揺 | 「なんか今、声が変わったのだ……？」 |
| 犯人が怒鳴った | 「な、なの……この人、怖いのだ……」 |
| 犯人が的外れな話をした | 「今の話、聞かれてないことまで話してるのだ？」 |
| プレイヤーが良い質問 | 「おおっ、なの！ いい質問なのだ！」 |
| プレイヤーが的外れ | 「うーん、ボクも気になるけど、事件と関係あるのだ？」 |

これらはトイマルの口調（一人称「ボク」、語尾「〜なのだ」）で統一する。

### トイマルの表情

尋問中のトイマルアイコンは状況に応じて変化する。

| 状態 | 表情 | 使用画像 |
|------|------|---------|
| 通常 | normal | `toimaru/event_normal.png` |
| 驚き・発見 | eureka（新規） | `toimaru/event_eureka.png` |
| 怖い | scared（新規） | `toimaru/event_scared.png` |
| 嬉しい | happy | `toimaru/event_happy.png` |
| 困惑 | confused | `toimaru/event_confused.png` |

`eureka` と `scared` は新規生成が必要。`docs/design_skills.md` のイベントモード画風ルールに従うこと。
