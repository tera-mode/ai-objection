# AI Objection プレイテスト指示書：尋問体験の検証と自律修正

> **目的**: プレイヤーが「何も知らない状態」から犯人とトイマルとのやりとりだけで推理を進め、クリアまで辿り着けるか——そしてその過程で**「自分の力で嘘を暴いた」という快感**を感じるかを検証する。
>
> **最終ゴール**: クリアできるかどうかではなく、**このゲームのコアな楽しさ（＝問う力で嘘が壊れる体験）を感じるかどうか**。
>
> **対象ケース**: case_001〜case_008 + mini_prologue（各ケース個別に実施）

---

## 第1章　顕在課題と根本原因

### 1-1. 症状

プレイヤーが犯人の発言とトイマルの発言から推理した質問をぶつけても、**次につながるキーワードが見つけられず、お手上げになる**。

### 1-2. 根本原因の仮説（4層構造）

```
Layer A: CriminalAI の返答に「次の糸口」がない
  → 犯人が否定するだけ/はぐらかすだけで、新しい情報を漏らさない
  → 「誘導キーワードルール」が機能していない

Layer B: トイマルのチップが「次の行動」に結びつかない
  → チップが犯人の既出キーワードの繰り返しにすぎない
  → チップを渡しても証拠がアンロックされない（トリガー不発）

Layer C: アンロックされた証拠の使い方がわからない
  → 証拠テキストから「何を質問すればいいか」が導けない
  → 証拠を突きつけても JudgeAI が矛盾を検出しない

Layer D: キーワード自体がプレイヤーの入力障壁になっている
  → 変換で出てこない漢字（例:「輝石水」「眠り蔦」）をプレイヤーが
    タイプできない／タイプする気にならない
  → 読み方がわからない固有名詞を質問文に組み込めない
  → 結果として「わかっているのに聞けない」という最悪のフラストレーション
```

このテストでは **Layer A→D の全層にわたり「連鎖の断絶ポイント」を特定し、修正する**。

### 1-3. 修正の大原則：壊して作り直してよい

> **既存の証拠品・キーワード・推理の流れは「聖域」ではない。**
> プレイヤーの体験を良くするためなら、遠慮なく壊して新しく作り直すこと。

```
❌ やってはいけないこと:
  「輝石水」というキーワードを維持するために、
  プレイヤーに無理やりタイプさせる工夫を考える

✅ やるべきこと:
  「輝石水」が入力しづらいなら、証拠品の名前自体を
  「光る水」「ひかり水」など打ちやすい名前に変える。
  それに伴って推理の流れが変わるなら、流れごと再設計する。
```

この原則は全問題（A〜F）の修正に適用される。
「既存設計を温存しようとして体験が悪化する」ことが最も避けるべき事態。

## 第3章　テスト手順

### Phase 0: 環境準備

```
1. https://ai-objection.vercel.app にアクセス
2. ゲストログイン
3. 歯車アイコン →「はじめから」でリセット
4. テスト対象のケースを選択
```

### Phase 1: 初期情報の記録（尋問開始前）

事件概要画面（crime-scene）で、以下を記録する。

```
記録項目:
  A. 事件概要テキストから読み取れる事実（箇条書き）
  B. 証拠スロットの数と、名前が見えるものの一覧
  C. 容疑者の紹介文から読み取れる印象
  D. 「最初に何を聞こうか」の候補（自然に思いつく質問3つ）
```

**判定基準**: Dが3つ思いつかない場合 → intro / criminalIntro のテキストが情報不足。

### Phase 2: 尋問ループ（コアテスト）

以下のループを**必ずクリア（逮捕成功）またはターン切れによる失敗まで完走**する。途中でテストを打ち切ってはいけない。1ターンごとに記録を取る。

```

│     - 変換で出てこない造語・固有名詞があった → Yes       │
│     - 読み方がわからない漢字があった → Yes               │
│     - スマホで5秒以内にタイプできない単語があった → Yes   │

│                                                       │

│   → 記録: 返答テキスト全文                              │
│   → 評価A': 返答は質問のトピックに正面から向き合っているか│
│             （肯定・否定・はぐらかし、いずれかの形で      │
│             質問の話題に言及していればYes）               │
│             Yes/No → Noのとき【すれ違いフラグ】ON        │
│             ⚠️ 質問と全く別の話をしていたらNo            │
│   → 評価A: 返答に「次の質問の糸口」となるキーワードが    │
│            含まれているか？ Yes/No + 該当キーワード      │
│   → 評価B: 動揺メーターは変化したか？ 変化量            │

│ STEP 3: トイマルのコメントを読む                        │
│   → 記録: コメントテキスト全文                          │

│                                                       │
│ STEP 4: トイマルパネルを開く                            │
│   → 記録: 表示されたチップの一覧                        │
│   → 評価D: どのチップを選ぶのが自然か？                  │
│   → チップを1つ選んでトイマルに渡す                      │
│   → 結果: 証拠アンロック有無 + トイマルの反応テキスト     │
│   → チップ未アンロックのとき: 「自分で入力」から          │
│     犯人の直前の発言中の重要単語を手動入力してみる         │
│     （「手すり」「議事録」「鐘」等、チップに出なかった語）  │
│     結果を記録: 手動入力アンロック 有 / 無                │

│   → 評価E: この証拠から「犯人への次の質問」が思いつくか？ │
│            Yes/No + 思いついた質問                      │
│                                                       │
│ STEP 6: 次のターンの質問を決める                        │

【オフパス質問テスト（必須）】
ターン2〜4のいずれか1回、以下を意図的に実行する:
  → 証拠なしの状態で、事件に関係しそうな「秘密の場所・物・行動」を
    直接プレイヤーが思いつく形で聞く（正規チップルートを外れた質問）
  → 例:「3階に何か特別な場所はありますか？」
       「被害者のノートを見ましたか？」
  → 評価A'（すれ違いフラグ）を必ず記録する
  → 目的: CriminalAIが「質問を無視して別の話を始める」問題Gを検出する

### Phase 3: 結果の集計と診断

全ターン完了後、以下を集計する。

```
■ すれ違いフラグ集計（最優先確認）
  → ONになったターン番号を列挙
  → 1件でもONなら即【問題G】として診断へ進む
  ⚠️ すれ違いは「行き詰まりフラグ」が立たなくても発生する。
     犯人が返答しているように見えて会話が成立していないパターン。

■ 証拠アンロック率集計（終了条件の一つ）
  → 総証拠数 / アンロック数 を集計
  → 8割未満の場合: 【問題C】の追加調査が必要
  → 確認ポイント:
     a. 自動チップ経由でアンロックされた証拠数
     b. 手動入力（「自分で入力」）でアンロックされた証拠数
     c. 一度も自動チップに出なかったが、手動入力で取れた証拠
        → trigger_keywords の見直しが必要（問題C）
     d. 手動入力しても取れなかった証拠
        → 犯人応答にそのトピックが自然に出るかを確認

■ トイマル自動チップ確認（手動補完が必要な証拠の特定）
  → 各証拠の trigger_keywords と、実際の自動チップを照合
  → 自動チップに出なかった重要キーワードがあれば記録する
    （初見プレイヤーが手動入力に気づかないリスク = 問題C 亜種）
  → 修正案: companionMemory のキーワードを犯人の実際の口調に合わせる

■ 行き詰まりフラグ集計
  → ONになったターン番号を列挙
  → 連続2ターン以上ONの箇所を「致命的行き詰まり」とマーク

■ 入力障壁フラグ集計
  → ONになったターン番号 + 該当キーワードを列挙
  → タイプしづらいキーワード一覧を作成（後で一括リネーム対象）

■ 情報連鎖の追跡
  各ターンの「次の質問の情報源」を追跡し、連鎖を図示する:

    [事件概要] → Q1 → [犯人返答のキーワードX]
                       → Q2 → [トイマルチップY → 証拠Z アンロック]
                              → Q3 → [証拠Zの内容から導いた質問]
                                     → ...

  連鎖が途切れている箇所 ＝ 「断絶ポイント」

■ コアファン判定
  以下のうち3つ以上Yesなら「コアな楽しさを感じた」と判定:
    □ 犯人の返答を読んで「こいつ嘘ついてる」と感じた瞬間があった
    □ 自分の質問で動揺メーターが動いた瞬間に快感があった
    □ 証拠を突きつけて犯人の口調が変わった瞬間があった
    □ トイマルの情報から「なるほど！」と繋がった瞬間があった
    □ 犯人が崩壊する瞬間に達成感があった
```

## 第4章　断絶ポイントの診断と自律修正

### 4-1. 診断フロー

行き詰まりが発生した各ターンについて、以下の診断を順に実行する。
**ここからはJSON・プロンプトを参照してよい。**

```
診断0（全ターン共通）: 入力障壁フラグが ON のキーワードはあるか？
  ├─ ある → 【問題F】キーワードがプレイヤーの入力障壁になっている
  └─ ない → 診断0Gへ
  ※ 問題F は他の問題と独立して常にチェックする

診断0G（全ターン共通）: すれ違いフラグが ON のターンはあるか？
  ├─ ある → 【問題G】CriminalAIが質問を無視して別の話をしている
  └─ ない → 診断1へ
  ※ 問題G は他の問題と独立して常にチェックする。問題A〜Eより優先度が高い。
    すれ違いが起きている状態では A〜E の診断が意味をなさない。

診断1: CriminalAI の返答に誘導キーワードがあるか？
  ├─ ある → 診断2へ
  └─ ない → 【問題A】CriminalAI が糸口を漏らしていない

診断2: トイマルのチップに事件関連のキーワードが含まれるか？
  ├─ ある → 診断3へ
  └─ ない → 【問題B】CompanionAI のチップ生成が的外れ

診断3: チップをトイマルに渡したとき、証拠がアンロックされたか？
  ├─ された → 診断4へ
  └─ されない → 【問題C】trigger_keywords が犯人の発言と噛み合っていない

診断4: アンロックされた証拠から次の質問が思いつけるか？
  ├─ 思いつける → 診断5へ
  └─ 思いつけない → 【問題D】evidence.content が推理の手がかりとして機能していない

診断5: 証拠を使った質問をぶつけたとき、JudgeAI が矛盾を検出したか？
  ├─ 検出した → 問題なし（別のターンの行き詰まりを調査）
  └─ 検出しない → 【問題E】JudgeAI の判定基準が strict すぎる / 
                          structural_weaknesses の記述が不十分
```

### 4-2. 問題別の自律修正アクション

#### 【問題G】CriminalAI が質問を無視して別の話をしている

```
症状: プレイヤーが「3階に隠し部屋がある？」と聞いたのに
      「1階で通常の業務をしていた」と全く別の話を返す

根本原因:
  システムプロンプトの「カバーストーリーの範囲で答えること」が
  「質問を無視してカバーストーリーを語る」と解釈される。
  または「知らない逃げを使わない」指示が強すぎて、
  否定の言葉すら使えず沈黙してカバーストーリーに戻る。

確認:
  a. src/lib/prompts/criminalAI.ts の以下を確認:
     - 秘密項目への返答指示（「カバーストーリーの範囲で答えること」付近）
     - proofLevel:'none' 時の指示
     - 「知らない逃げを使わない」指示
  b. 問題のターンで犯人が「質問のトピックに1文字も触れていないか」を確認

修正案:
  1. 秘密項目の返答指示を「向き合ってからはぐらかす」形に修正:
     ❌「カバーストーリーの範囲で答えること」
     ✅「必ず質問の話題に正面から向き合い（「何の話じゃ？」「そんなことは……」）
        否定・はぐらかしをしてから別の話に移ること。質問を無視してはいけない」

  2. proofLevel:'none' 指示に「質問無視禁止」を追記:
     「自信を持って否定してよい。ただし、必ず今回の質問の話題（場所・行動・物）に
      言及した上で否定すること。質問を無視して別の話をしてはいけない」

  3. 「知らない逃げ禁止」を緩和:
     ❌「「知らない」という逃げを使わない」
     ✅「「思い出せない」「記憶にない」という逃げは使わない。
        ただし「何の話じゃ？」「そんな場所は知らん」という直接否定は使ってよい」
```

#### 【問題A】CriminalAI が糸口を漏らしていない

```
原因: 犯人が完璧に嘘をつき、新しい情報を一切漏らしていない
確認: criminalAI.ts の「誘導キーワードルール」が systemPrompt に含まれているか

修正案:
  1. criminalAI.ts の【演技の細則】に以下を強化:
     「返答の中に、カバーストーリーの別の部分に触れるキーワードを
      必ず1つ含めること。例: 場所の嘘をつくとき、時間の矛盾に
      繋がる言葉をうっかり混ぜる」

  2. ケースJSON の coverStory.claim を確認:
     claim の中に「後で崩れる伏線キーワード」が含まれているか？
     含まれていなければ claim を修正して伏線を埋め込む。

  3. personality.surface / underPressure に
     「ボロを出すパターン」を具体的に追記:
     例: 「プレッシャーを感じると、聞かれていない時間帯の
          行動をうっかり話してしまう」
```

#### 【問題B】CompanionAI のチップ生成が的外れ

```
原因: チップが犯人の返答の表層的な単語の反復になっている
確認: /api/companion-chips のプロンプトを確認

修正案:
  1. companion-chips API のプロンプトに以下を追加:
     「チップは犯人の返答から"不自然な点"や"深掘りすべき点"を
      抽出すること。単なるキーワードの繰り返しではなく、
      『なぜ？』『本当に？』と思える切り口を提示すること」

  2. チップに「行動指示ニュアンス」を含める:
     ❌ 「薬草園」（単語だけ）
     ✅ 「薬草園のこと」（→トイマルに渡すと証拠が出そうな予感がする）
     
  3. チップの表示順を「事件関連度が高い順」にソートする
```

#### 【問題C】trigger_keywords が犯人の発言と噛み合っていない

```
原因: companionMemory の trigger_keywords が狭すぎる / 
      犯人が想定キーワードを使わない言い回しをしている

確認: 
  a. 対象の companionMemory エントリの trigger_keywords を確認
  b. 実際の犯人の返答テキストと照合

修正案:
  1. trigger_keywords を拡充:
     - 同義語・類義語を追加（例: 「薬草」→「薬草, 植物, 草, 園」）
     - 犯人の口調で出やすい言い回しを追加
     - 部分一致でもヒットするようにする

  2. companion-trigger API の判定ロジックを確認:
     - 完全一致ではなく、意味的な近さでも判定しているか？
     - 犯人の発言中にキーワードの関連語があれば
       トリガー条件「犯人が関連トピックに言及済み」を満たすか？

  3. ケースJSON側の修正:
     - trigger_keywords に犯人の personality.surface で
       使いそうな言葉を追加する
```

#### 【問題D】evidence.content が推理の手がかりとして機能していない

```
原因: 証拠テキストが事実の列挙で、「何が怪しいか」の
      手がかりがプレイヤーに伝わらない

確認: 該当の evidence.content を読み、中学生が
      「犯人に何を聞けばいいか」を思いつけるかシミュレーション

修正案:
  1. 証拠の「声」の中に、犯人の主張と矛盾するポイントが
     明示的に含まれているか確認:
     ✅「持ち出しは眠り蔦の区画から1束のみ」
        → 犯人が「薬効成分入り」と言っていたのと矛盾する
     ❌「訪問記録があります」
        → それだけでは何が矛盾なのかわからない

  2. 証拠テキストの末尾に「声の出典（誰が誰に話していたか）」が
     書かれているか確認。出典がないと証拠の信頼性が伝わらない。

  3. 必要に応じて evidence.content を書き直す:
     - 犯人の主張（coverStory.claim）の具体的な一部と
       直接矛盾する事実を証拠テキストに含める
     - 「〜のみ」「〜だけ」「〜ゼロ」など、
       限定的な表現で矛盾を際立たせる
```

#### 【問題E】JudgeAI の判定基準が strict すぎる

```
原因: プレイヤーの質問が「正解の言い回し」と少しずれているだけで
      矛盾と判定されない

確認: 
  a. judgeAI.ts のプロンプトで structural_weaknesses が
     どう参照されているか確認
  b. 問題のターンで JudgeAI に渡されたプレイヤーの質問文と
     structural_weaknesses の対応を確認

修正案:
  1. structural_weaknesses の記述を、より広い表現に修正:
     ❌「ev_garden_log が提示されたとき」（提示方法が限定的）
     ✅「プレイヤーが薬草園の訪問記録・持ち出し内容に言及したとき」

  2. judgeAI.ts のプロンプトに「意味的な近さで判定せよ」を追加:
     「プレイヤーの質問が structural_weaknesses の
      キーワードと完全一致しなくても、意味的に同じ矛盾を
      指摘していれば has_contradiction: true とせよ」

  3. 証拠の突きつけ判定を緩和:
     - 証拠IDの明示的な引用がなくても、
       証拠の内容に言及していれば proof_level を上げる
```

#### 【問題F】キーワードがプレイヤーの入力障壁になっている

```
原因: ゲーム内の固有名詞・証拠名・キーワードが、スマホの日本語入力で
      変換しづらい or 読み方がわからない

確認:
  a. 入力障壁フラグが ON のキーワード一覧を確認
  b. 各キーワードをスマホの日本語キーボードで実際にタイプしてみる
  c. 以下のいずれかに該当すれば NG:
     - IME変換の候補に出てこない造語（例:「輝石水」「眠り蔦」）
     - 読み方が推測できない漢字（例:「煌めく」→「きらめく？」）
     - 4文字以上の固有名詞で変換に2ステップ以上必要

判定基準（NGキーワード）:
  スマホで「あ、これ面倒だな」と一瞬でも思ったら NG。
  中学生は「面倒」と感じた瞬間にやめる。

修正の原則:
  ⚠️ キーワードをそのまま維持しようとしないこと。
  プレイヤーの体験が最優先。名前を変えることで推理の流れが
  変わるなら、推理の流れごと作り直す。

修正案:
  1. 固有名詞のリネーム:
     ❌「輝石水」→ ✅「ひかり水」「光の水」
     ❌「眠り蔦」→ ✅「ねむり草」「眠り草」
     ❌「メキーラ」→ そのまま（カタカナは変換不要でOK）

  2. リネームに伴う連鎖修正:
     - evidence[].name / evidence[].content 内の表記を一括置換
     - coverStory.claim 内の該当キーワードを置換
     - companionMemory[].trigger_keywords を新名称に更新
     - structural_weaknesses 内の参照を更新
     - storyText（intro / criminalIntro / victory / defeat）を更新
     - timeline 内の表記を更新
     - criminal.knowledge.knows / doesnt_know 内を更新

  3. キーワード命名の新ルール:
     - ひらがな or カタカナ or 小学3年生までの漢字で書けること
     - スマホで一発変換できること
     - 2〜4文字が理想（長すぎも障壁）
     - 世界観の雰囲気を壊さない範囲で「打ちやすさ」を優先

  4. 証拠品名の特別ルール:
     証拠品はプレイヤーが質問文に入れて使うため、最も打ちやすさが重要。
     「〜の声」形式は維持するが、中の固有名詞を平易にする。
     例:「品質検査人の声」→「調べた人の声」
        「薬草園管理人の声」→「薬草の園の人の声」「草の管理人の声」
```

## 第5章　「コアファン」の成立条件

### 5-1. このゲームの「コアな楽しさ」の定義

```
A. 発見の快感:「こいつ、今おかしなこと言った！」
   → 犯人の返答に「嘘っぽさ」を自分で見つける瞬間

B. 追い詰める快感:「この証拠があるんだけど？」
   → 証拠を突きつけて動揺メーターが動く瞬間

C. 崩壊の快感:「やった、ボロ出した！」
   → 犯人の口調が変わり、嘘が崩れていく過程

D. 成長の実感:「最初は何もわからなかったのに……」
   → ゼロの状態から自力で真実に辿り着いた達成感
```

### 5-2. コアファンが成立しない典型パターン

```
パターン①「壁打ち」
  何を聞いても犯人が「知りません」「おかしいですね」で返す
  → 動揺メーターが一切動かない → A, B, C すべて不成立
  → 根本原因: 問題A + 問題E

パターン②「空回り」
  トイマルのチップを渡しても証拠がアンロックされない
  → 証拠がないので突きつけられない → B 不成立
  → 根本原因: 問題C

パターン③「手がかり迷子」
  証拠はアンロックされたが、そこから何を聞けばいいかわからない
  → 証拠を持っているのに使えない → D 不成立
  → 根本原因: 問題D

パターン④「すり抜け」
  正しい質問をしているのに JudgeAI が矛盾を検出しない
  → 「合ってるはずなのに効かない」→ ストレス → A〜D すべて阻害
  → 根本原因: 問題E

パターン⑤「すれ違い」
  何を聞いても犯人が質問と無関係な話を返し続ける
  → 「このAI、ちゃんと聞いてる？」という不信感でゲームへの没入が崩壊
  → 動揺メーターも動かないのでさらに絶望感
  → 根本原因: 問題G
  ⚠️ このパターンは「行き詰まりフラグ」が立ちにくい（犯人は返答しているため）
     テスターが「会話が成立しているか」を意識的に確認しないと見逃す
```

### テスト推奨順序

```
1. mini_prologue  — チュートリアル。ここが壊れていると全体が破綻
2. case_001       — 最初の本格ケース。easy の基準線

## 第10章　自律修正の実行フロー（AIテスター向け）

このセクションは、Claude Code 等のAIエージェントがプレイテストから修正までを自律的に実行するための手順。

### Step 1: テスト実行

```
1. 対象ケースの crime-scene 画面を開く
2. Phase 1 の情報を記録する
3. Phase 2 のループをクリア（逮捕成功）またはターン切れ（失敗）まで必ず完走し、
   各ターンのテンプレートを埋める
   ⚠️ 途中打ち切り禁止。結果画面（逮捕成功 or 敗北）が出るまでテストを続けること
4. Phase 3 の集計を行う
```

### Step 2: 問題の診断

```
1. すれ違いフラグが ON のターンを抽出 → 問題G に分類（最優先）
   すれ違いがある状態で他の問題を診断しても意味がない。先に問題Gを解消する。
2. 入力障壁フラグが ON のキーワードを抽出 → 問題F に分類
3. 行き詰まりフラグが ON のターンを抽出
4. 各ターンについて第4章の診断フローを実行
5. 問題 A〜G に分類
```

### Step 3: 修正の実行

**⚠️ 修正の大原則（再掲）**: 既存の証拠品・キーワード・推理の流れは
「聖域」ではない。体験改善のためなら壊して作り直してよい。

```
問題F の場合（最優先で実行。他の問題より先に片付ける）:
  → 入力障壁キーワード一覧を確認
  → 各キーワードのリネーム案を作成（ひらがな/カタカナ/簡単な漢字）
  → data/cases/case_XXX.json 内の全箇所を一括置換:
     evidence[].name, evidence[].content, coverStory.claim,
     companionMemory[].trigger_keywords, structural_weaknesses,
     storyText, timeline, criminal.knowledge.knows/doesnt_know
  → リネームにより推理の流れが成立しなくなる場合:
     推理の流れ自体を再設計する（evidence, weaknesses, claim を書き直す）
  → 置換後、ケース制作ガイドの完成前チェックリストで整合性を確認

問題A の場合:
  → data/cases/case_XXX.json の coverStory.claim を確認
  → claim に伏線キーワードが不足していれば追加
  → personality に「ボロを出すパターン」を追記

問題B の場合:
  → /api/companion-chips のプロンプトを確認・修正

問題C の場合:
  → data/cases/case_XXX.json の companionMemory[].trigger_keywords を拡充
  → 犯人の実際の返答で使われた言い回しを trigger_keywords に追加

問題D の場合:
  → data/cases/case_XXX.json の evidence[].content を修正
  → 犯人の claim と直接矛盾するフレーズを証拠テキストに含める

問題E の場合:
  → data/cases/case_XXX.json の structural_weaknesses の記述を広げる
  → src/lib/prompts/judgeAI.ts の判定基準を確認・修正
```

### Step 4: 修正後プレイテスト（PDCAのC）

```
⚠️ 修正した内容は、必ず再度プレイテストで検証する。
   「修正した＝完了」ではない。「検証して問題がなくなった＝完了」。

1. 修正した箇所に関連するターン周辺のみ再テスト（部分テスト）
2. 行き詰まりフラグが OFF になったことを確認
3. 全断絶ポイントが解消されたら、通しプレイで最終確認（全ターン完走）
4. 最終確認でコアファン判定 4/5 以上を目標とする
```

### Step 5: PDCAループの判定と継続

```
再テストの結果をもとに以下を判断する:

■ 新たな問題が発見された場合 → ループを続ける
  → Step 2（診断）に戻る
  → 新たな断絶ポイントを診断・分類
  → Step 3（修正）→ Step 4（再テスト）を繰り返す

■ 全ての問題が解消された場合 → ループを終了してStep 6へ

PDCAサイクルのイメージ:

  Plan   → このケースのテスト目標と確認観点を設定（第8章）
  Do     → Step 1 でプレイテストを実行
  Check  → Step 2〜3 で問題を診断し修正
  Act    → Step 4 で修正を検証
  ↑_____________________________↓（問題が残っていれば繰り返す）

終了条件（これが満たされるまでループする）:
  □ すれ違いフラグ ON のターンがゼロ（最優先）
  □ 行き詰まりフラグ ON のターンがゼロ
  □ 入力障壁フラグ ON のキーワードがゼロ
  □ 証拠アンロック率 8割以上（総証拠数 × 0.8 個以上をアンロックして逮捕）
  □ コアファン判定 4/5 以上
  □ クリア（逮捕成功）していること
```

### Step 6: レポート出力

```
修正前後の比較を含むレポートを出力:
  - 修正前の行き詰まりターン → 修正後の結果
  - 修正したファイルと変更内容の diff（コミットSHAを含む）
  - コアファン判定の変化（修正前 X/5 → 修正後 Y/5）
  - PDCAのループ回数（何周で解消されたか）
```
